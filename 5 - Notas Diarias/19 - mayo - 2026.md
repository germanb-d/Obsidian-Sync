# Modelo
El modelo es una representación abstracta del mundo real. Son abstracciones, supuestos y restricciones en el dominio.

**Beneficios**
- Establece una línea de base de comprensión.
- Facilita la comunicacion entre los stakeholders.
- Permite análisis formales.
- Simplifican el dominio de aplicación.

Tipo de Problema
Empíricos: Desarrollar una especificación requiere un estudio constante del mundo real. Ej: En la metereologia.

Formales: Abstracción, estructura y representación del conocimiento en una forma que permita razonar sobre él. Ej: En la abstraccion para logica matematica.

Ingenieriles: Implementación de la construcción establecida por los principios aceptados. Ej: Nomenclatura para un plano de construcción.

Osea el modelo conceptual: es el "que es" del problema y no el "como se resuleve". No se ve la implementacion para una solucion.

### Propiedades de modelos conceptuales 

Constructibilidad: Permitir la comunicación usuario-analista que se acomode al manejo de gran número de datos que maneje el problema de la complejidad fácil de analizar, si es ambiguo, incompleto o inconsistente

Ejecutabilidad: Es importante en la validación de la especificación y está sujeta al formalismo
que se use.
%%> Es poder hacer una traza entre nuestro modelo conceptual del problema y aquela solucion planteada en la solucion %%
Traceability: Existencia de un cross reference que conecte la especificación conceptual con la especificacion del diseno e implementación 

Minimalidad: Cada concepto tiene un significado distinto y ningún concepto puede ser
expresado en términos de otros.

- **NFR Name**: el nombre del requisito no funcional.  
    Ej: “Seguridad”, “Rendimiento”.
- **Description**: una explicación corta de qué exige ese RNF.  
    Ej: “El sistema debe responder rápido”.
- **Design Decisions**: decisiones técnicas tomadas para cumplir ese RNF.  
    Ej: usar caché para mejorar velocidad.
- **Level of Priority**: qué tan importante es el RNF, de 0 a 10.  
    10 = crítico, 0 = irrelevante.
- **General Attributes**: restricciones o comportamientos del sistema en ejecución.  
    Ej: tiempo máximo de respuesta de 2 segundos.
- **Data Attributes**: características de los datos necesarias para cumplir el RNF.  
    Ej: datos encriptados, tamaño máximo de archivos.


# Validación de requerimientos

"Proceso que certifica que se ataca el problema correcto"

Hay enfoques de Ingeniería de Requerimientos que no lo separan de elicitación y especificación.
La diferenciación es importante por razones conceptuales y prácticas.


## Etapas

**Propósito**
- certifica la consistencia del modelo (de requerimientos) con las intenciones de clientes y usuarios.
- más general que la idea común de validación la necesidad aparece cuando se modifica el modelo actual.
- se aplica también a los modelos intermedios.

**Input**
- todo modelo está sujeto a validación, por lo tanto cada modelo es input.
- el conocimiento sobre el dominio del problema.
- algunas partes del modelo formal.

**Actividades**
- interacción entre analistas, clientes del sistema y usuarios en el dominio del problema.
- similar a formular una nueva teoría científica y posteriormente testearla.


![[19 - mayo - 2026-1779213719775.webp]]

"No significa que el modelo sea correcto": 
Puede que haya expectativas que aunque el modelo este bien, este puede no ser correcto porque estas expectativas no son factibles en la realidad. (por tecnologia, presupesto, etc)
